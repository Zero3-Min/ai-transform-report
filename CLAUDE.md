# 项目工作规范

## 项目目标
产出一份面向 10–200 人网络传媒公司（MCN、内容工作室、代运营、地方新媒体）的
AI 转型咨询报告。读者是公司老板和主编：语言专业但拒绝黑话，每个论断都要
回答"关我什么事、要花多少钱、多久见效"。

## 目录结构（严格遵守）
- raw/       原始底稿与客户资料。只读，任何任务不得修改此目录。
- research/  调研笔记。每个主题一个 md 文件，编号前缀（01- 02- ...）。
- cases/     案例库。每个案例一个 md 文件。
- report/    报告正文。每章一个文件（01-executive-summary.md ... 13-appendix.md）。
- review/    审查报告（review-facts.md、review-biz.md）。
- final/     定稿输出。

## 数据可靠性规则（最高优先级）
1. 所有事实性数据（百分比、金额、日期、政策条款）必须紧跟来源标注，
   格式：[来源: 机构/媒体名, 《标题》, 发布日期, URL]
2. 搜索不到可靠来源的数据：不得写入正文，统一放入该文件末尾"## 待核实"小节。
3. 来源优先级：政府文件/法规原文 > 头部咨询机构报告(麦肯锡/德勤/艾瑞/QuestMobile)
   > 上市公司财报/官方通稿 > 主流媒体报道 > 自媒体（自媒体仅可作观点引用，不可作数据来源）。
4. 引用第三方内容一律用自己的话转述，单一来源直接引用不超过一句短语。
5. 政策类内容（如 AIGC 标识办法）必须核对法规原文，不得转引二手解读的条款表述。

## 案例文件固定结构
公司背景 / 做了什么 / 用了什么技术 / 量化效果（无公开出处则标"该数字未经证实"）
/ 对 10–200 人小公司的适配启示 / 来源链接

## 写作与一致性
- 修改任何章节前，先读 report/README.md 目录索引和相邻章节，保持术语与数据口径一致。
- 全文统一术语表：知识库（不写"知识中台"）、创作流水线、预审 Agent、人工终审、陪跑服务。
- 报价与 ROI 数字必须与 report/ 内其他章节互相一致，改一处需全局检查。

## Git 规范
- commit message 用中文，格式：`[章节/模块] 做了什么`
- 每个任务的产出必须全部 commit，不留未跟踪文件。

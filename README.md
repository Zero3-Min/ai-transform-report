# 网络传媒公司 AI 转型咨询报告

面向 **10–200 人网络传媒公司**（MCN、内容工作室、代运营、地方新媒体）的 AI 转型咨询报告完整工作仓库。读者是公司老板和主编，全文遵循一个写作标准：每个论断都要回答**"关我什么事、要花多少钱、多久见效"**。

## 仓库结构

```
├── raw/        原始底稿与客户资料（只读，不修改）
├── research/   调研笔记（4 份，含来源标注与待核实清单）
├── cases/      案例库（5 个真实公开案例 + 索引）
├── report/     报告正文（13 章 + 目录索引，核心交付物）
├── review/     审查报告（事实审查 + 商业审查，质量证明）
├── final/      定稿输出（交付客户的最终版本存放处）
└── CLAUDE.md   项目工作规范（数据可靠性规则、术语表、Git 规范）
```

### 各目录说明

| 目录 | 内容 | 性质 |
|---|---|---|
| `raw/` | 客户提供的原始方案底稿 | 内部资料，溯源锚点 |
| `research/` | 01-行业数据、02-政策合规、03-技术选型、04-报价对标 | 内部工作档案，支撑正文数据 |
| `cases/` | 新华智云、界面财联社、瑞安融媒、天下秀、遥望科技 | 可随报告附送客户 |
| `report/` | 01-executive-summary 至 13-appendix，共 13 章 | **核心交付物** |
| `review/` | review-facts.md（事实审查）、review-biz.md（商业审查） | 质量证明，可选择性展示 |
| `final/` | 定稿输出 | 面向客户的最终交付版本 |

### 报告正文 13 章一览

导航与各章关键数字清单见 [report/README.md](report/README.md)。章节依次为：执行摘要 → 行业背景 → 成熟度诊断 → 转型蓝图与场景 → 知识库与创作流水线 → 技术架构选型 → 合规与风控 → 组织变革 → 实施计划（WBS）→ 效果度量与 ROI → 报价与商务模式 → 风险登记册 → 附录（诊断清单 + 验收指标表）。

## 数据可靠性承诺

- 所有事实性数据（百分比、金额、日期、政策条款）**紧跟来源标注**，格式为 `[来源: 机构/媒体名, 《标题》, 发布日期, URL]`，来源优先级：政府文件 > 头部咨询机构 > 上市公司财报 > 主流媒体（自媒体不作数据来源）。
- 搜索不到可靠来源的数据一律放入各文件末尾的"待核实"小节，**不写入正文**。
- 无法用外部数据替换的预测性数字（报价、ROI、验收线）统一标注为基于项目经验的估算，签约前按客户实况校准。
- 政策条款（如《人工智能生成合成内容标识办法》）均核对法规原文表述，不转引二手解读。
- 全文已通过两轮独立审查（事实审查 + 商业审查），发现的 18+ 项问题已全部修复，问题清单与修复记录完整保留在 `review/` 目录。

## 如何展示给客户

### 交付物分层

- **直接交付**：`report/` 全部 13 章（或整合后放入 `final/` 的单文件版本）+ `cases/` 案例库。
- **按需展示**：`review/` 两份审查报告——当客户询问"报告里的数字可靠吗"时，主动出示审查记录是最有力的信任背书（这也是本仓库保留审查痕迹的目的）。
- **不对客户展示**：`raw/`（客户原始资料）、`research/`（内部调研档案，含未采信数据的排查过程）、`CLAUDE.md`（内部工作规范）。

### 建议的展示流程（一次客户会议的走法）

1. **开场用第 1 章执行摘要**（5 分钟）：三个核心判断——"转型不是要不要做而是怎么做"、"先建知识库再上生成工具"、"人工终审不可妥协"——先让老板建立整体认知。
2. **用第 2 章回答"关我什么事"**（5 分钟）：行业近八成机构利润率下滑的经营压力数据，把 AI 转型从"新技术尝鲜"落位为"守住利润率的经营手段"。
3. **用第 4 章场景优先级表锁定切入点**（10 分钟）：P0"不做会出事"（合规问答）、P1"早做早受益"（新人培训、初稿生成），现场和客户圈出他们最痛的 2–3 个场景。
4. **用第 10、11 章回答"花多少钱、多久见效"**（10 分钟）：报价四档表 + ROI 测算方法演示（强调数字会用客户真实数据重算并写入合同），静态回收期预期 < 2 个月。
5. **用第 7 章合规检查清单收尾**（5 分钟）：AIGC 标识新规的执法数据（一次专项行动处置 1.3 万个账号）+ 四平台规则对比表，这是同类方案少有的差异化内容，也是留下"这家顾问真懂传媒行业"印象的关键。
6. **案例随问随答**：客户问"有没有人做成过"时按对方体量选案例——大机构问就讲界面财联社（三审后产出提升 3 倍），小团队问就讲瑞安融媒（县级机构快速接入），客户过度乐观时讲遥望科技（技术投入不等于商业成功）。

### 不同角色给不同材料

| 客户角色 | 建议材料 |
|---|---|
| 老板 / 决策人 | 第 1、2、10、11 章 + 遥望科技警示案例 |
| 主编 / 内容负责人 | 第 3、5、8 章 + 界面财联社案例 |
| 技术负责人（如有） | 第 6 章（开头已附"给老板的一句话总结"，无技术负责人时老板只看结论即可） |
| 运营 / 合规专员 | 第 7 章 + 发布前合规检查清单（可直接打印上墙） |

## 工作流说明

本仓库按 `raw →（research + cases）→ report → review → final` 的流水线组织：原始底稿拆分为 13 章框架，四轮联网调研补充可核实数据，案例库提供实证支撑，两轮审查（挑剔的客户 CTO 视角 + 50 人 MCN 老板视角）完成质量把关。每一步的产出与修复均有独立 git 提交记录可追溯。
